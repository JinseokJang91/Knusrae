# 환경 변수 설정 가이드

OAuth·DB·JWT 등 민감한 값은 **환경 변수**로만 넣고, 코드/설정 파일에는 기본값을 두지 않는 것을 권장합니다.

---

## 0. 환경 변수 값은 어디에 두고, yml에는 어떻게 반영되나?

### 값을 저장하는 곳 (환경별)

| 환경 | 값 저장 위치 | 비고 |
|------|--------------|------|
| **로컬** | Run Configuration 환경변수, 터미널 `export`/`$env:`, 또는 `.env` 파일(IDE/플러그인으로 로드) | common-service는 단독 실행하지 않으므로, **auth-service / cook-service / member-service**를 실행할 때 그 Run Configuration에 변수를 넣으면 됨. |
| **개발 서버** | 해당 서버(또는 컨테이너)의 **시스템 환경 변수** 또는 배포 시 주입하는 **.env / 시크릿 파일** | 서버에 올릴 때 한 번만 설정. |
| **운영(prod)** | EC2/ECS/컨테이너/플랫폼의 **환경 변수** 설정, 또는 시크릿 매니저(AWS Secrets Manager 등)에서 주입 | 코드/설정 파일에는 넣지 않음. |

즉, **yml 파일 안에 비밀값을 적어두지 않고**, “실제로 앱을 실행하는 프로세스”에게만 환경 변수로 넘깁니다.

### yml에 반영되는 방식 (Spring Boot 동작)

1. **실행 주체**: 서버에 올리는 건 **auth-service**, **cook-service**, **member-service** 같은 애플리케이션입니다. common-service는 라이브러리(JAR)로 포함되어 함께 실행됩니다.
2. **설정 로드 순서**: 각 서비스가 기동할 때 `application.yml` → `application-common.yml`(import) → `application-{profile}.yml`(예: dev, prod) 순으로 설정을 읽습니다.
3. **치환 규칙**: yml 안의 `${JWT_SECRET}`, `${DB_URL}` 같은 **플레이스홀더**는, **그때 실행 중인 프로세스의 환경 변수**에서 같은 이름(`JWT_SECRET`, `DB_URL`)을 찾아서 치환합니다.
   - 환경 변수가 **있으면** → 그 값으로 치환됩니다.
   - **없고** `application-common.yml`처럼 `${JWT_SECRET:기본값}` 형태면 → 기본값이 사용됩니다(로컬/테스트용).
4. **별도 “환경변수 사용 설정” 불필요**: `application-common.yml`이나 다른 yml에서 “환경변수를 쓰도록” 따로 설정할 필요는 없습니다. **실행 시점에 프로세스에 환경 변수만 넘기면** Spring이 자동으로 `${변수명}`을 그 값으로 바꿉니다.

정리하면, **개발/운영에서 application-common.yml에 환경변수를 “세팅하도록” 하는 건**, yml을 수정하는 게 아니라 **해당 서비스(auth, cook, member)를 띄울 때 환경 변수로 값을 넘기는 것**입니다. 값은 위 표의 “저장 위치”에 두고, yml에는 `${변수명}`만 두면 됩니다.

**개발/운영에서 값을 넣는 예**

- **Docker**: `docker run -e JWT_SECRET=xxx -e DB_URL=yyy ...` 또는 `docker-compose.yml`의 `environment:` / `env_file:`
- **EC2 등 서버**: 서버에 `.env` 파일을 두고 systemd/스크립트에서 `export $(cat .env | xargs)` 후 jar 실행 (해당 파일은 git에 넣지 않음)
- **Kubernetes**: ConfigMap/Secret을 Pod의 `env` 또는 `envFrom`으로 주입
- **GitHub Actions**: workflow에서 `env: JWT_SECRET: ${{ secrets.JWT_SECRET }}` 로 주입 후 배포 스크립트가 서버/컨테이너에 전달

위처럼 **프로세스가 시작될 때** 환경 변수가 세팅되면, `application-common.yml`을 불러오는 그 같은 프로세스 안에서 `${JWT_SECRET}` 등이 자동으로 치환됩니다.

## 1. 로컬 개발 (내 PC에서 실행할 때)

**방법 A: IDE에서 설정 (권장)**

- **IntelliJ / Cursor**: Run Configuration → Edit Configurations → 해당 Spring Boot 앱 선택 → **Environment variables** 에 아래 변수 추가  
  - 예: `NAVER_CLIENT_ID=xxx`, `NAVER_CLIENT_SECRET=yyy` (한 줄에 하나씩 또는 `ENV_VAR=value` 형식)
- **VS Code**: `.vscode/launch.json` 의 `env` 에 변수 추가 (이 파일은 커밋하지 말고, `.vscode/launch.json` 이 gitignore 인지 확인)

**방법 B: 터미널에서 설정 후 실행**

- **Windows (PowerShell)**  
  ```powershell
  $env:NAVER_CLIENT_ID="your_naver_client_id"
  $env:NAVER_CLIENT_SECRET="your_naver_client_secret"
  $env:GOOGLE_CLIENT_ID="..."
  $env:GOOGLE_CLIENT_SECRET="..."
  $env:KAKAO_CLIENT_ID="..."
  $env:KAKAO_CLIENT_SECRET="..."
  # 그 다음 auth-service 실행
  ```
- **Mac / Linux (bash)**  
  ```bash
  export NAVER_CLIENT_ID="your_naver_client_id"
  export NAVER_CLIENT_SECRET="your_naver_client_secret"
  export GOOGLE_CLIENT_ID="..."
  export GOOGLE_CLIENT_SECRET="..."
  export KAKAO_CLIENT_ID="..."
  export KAKAO_CLIENT_SECRET="..."
  # 그 다음 auth-service 실행
  ```

**방법 C: 로컬 전용 설정 파일 (git 제외)**

- `backend/auth-service/src/main/resources/application-local.yml` 은 **이미 .gitignore 대상**입니다.
- `application-local.example.yml` 을 복사해 `application-local.yml` 로 만들고, 그 안에 **직접 값**을 넣어 사용할 수 있습니다.  
  (이 파일은 절대 커밋하지 마세요.)

**로컬에서 필요한 변수 (auth-service 기준)**

| 변수명 | 설명 |
|--------|------|
| `NAVER_CLIENT_ID` | 네이버 앱 Client ID |
| `NAVER_CLIENT_SECRET` | 네이버 앱 Client Secret |
| `GOOGLE_CLIENT_ID` | Google OAuth Client ID |
| `GOOGLE_CLIENT_SECRET` | Google OAuth Client Secret |
| `KAKAO_CLIENT_ID` | 카카오 앱 REST API 키 |
| `KAKAO_CLIENT_SECRET` | 카카오 앱 Client Secret |

## 2. GitHub (CI/CD·자동 배포)

- **실제 비밀값은 코드나 설정 파일에 넣지 않고**, **GitHub 저장소 설정**에서만 관리합니다.
- **Settings → Secrets and variables → Actions** 에서 저장소/환경별 **Secrets** 추가  
  (예: `NAVER_CLIENT_ID`, `KAKAO_CLIENT_SECRET` 등).
- GitHub Actions 워크플로우에서는 `secrets.NAVER_CLIENT_ID` 처럼 참조합니다.  
  → **GitHub는 “비밀값 보관 + CI에서 주입” 용도**이고, 로컬 PC의 환경 변수를 대체하는 것은 아닙니다.

---

## 3. 서버/배포 환경 (운영)

- 실제 서비스가 도는 서버(또는 컨테이너·PaaS)의 **환경 변수**에 위와 같은 이름으로 값을 설정합니다.
- 예: AWS EC2/ECS, Azure App Service, Docker run 시 `-e NAVER_CLIENT_ID=...`, Vercel/Netlify 환경 변수 등 각 플랫폼의 “환경 변수 설정” 메뉴를 사용하면 됩니다.
- **별도 환경설정 파일**을 서버에 두고 싶다면, 그 파일은 반드시 **git에 넣지 말고**, 배포 스크립트나 시크릿 매니저로만 서버에 전달하는 방식이 안전합니다.

---

## 요약

| 용도 | 설정 방법 |
|------|------------|
| **로컬 개발** | IDE Run Configuration 환경 변수 **또는** 터미널 `export`/`$env:` **또는** git 제외된 `application-local.yml` 에 직접 입력 |
| **GitHub** | 저장소 **Secrets and variables → Actions** 에만 저장, 워크플로우에서 `secrets.XXX` 로 사용 |
| **운영 서버** | 해당 서버/컨테이너/플랫폼의 **환경 변수** 설정 (별도 파일 사용 시 반드시 git 제외) |

**GitHub에 올리면 안 되는 것**: 실제 Client ID/Secret, DB 비밀번호, JWT 시크릿 등이 들어 있는 파일.  
**GitHub에 올려도 되는 것**: `application-local.example.yml` 처럼 **변수 이름·형식만 있는 예시 파일**, 그리고 이 가이드 문서.
