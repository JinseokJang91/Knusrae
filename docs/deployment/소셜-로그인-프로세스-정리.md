## 소셜 로그인 프로세스 정리 (NAVER / KAKAO / GOOGLE)

### 1. 전체 구조 개요

- **지원 프로바이더**: `NAVER`, `KAKAO`, `GOOGLE`
- **인증 방식**: 각 프로바이더의 OAuth 2.0 Authorization Code Grant 사용
- **토큰 저장 방식**: 백엔드에서 발급한 JWT `accessToken`, `refreshToken`을 **HttpOnly 쿠키**로 저장 (`CookieUtils` 이용)
- **클라이언트 상태 관리**: 토큰 내용은 프론트에서 직접 보지 않고, `authStore`(Pinia) + `/api/member/me`로 로그인 상태와 프로필 정보를 관리
- **콜백 처리 방식**: OAuth 콜백은 **별도 라우트 + 팝업 창**으로 처리하고, `window.postMessage` / `localStorage`를 이용해 부모 창과 통신

프론트/백엔드 간 책임 분리는 다음과 같다.

- **프론트엔드 (`frontend/`)**
  - 소셜 로그인 버튼 UI (`SocialLoginButtons`)
  - 각 프로바이더 OAuth 인증 URL 구성 및 팝업 오픈 (`openOAuthPopup`)
  - `/auth/{provider}/callback` 라우트 및 콜백 화면 (`OAuthCallback`)
  - 팝업에서 온 메시지를 수신하여 `authStore.login()` 호출 및 페이지 리다이렉트
- **백엔드 (`backend/auth-service/` + `backend/common-service/`)**
  - `/api/auth/{provider}/callback` 엔드포인트 처리 (`AuthController`)
  - 각 프로바이더 액세스 토큰 발급 및 사용자 정보 조회 (`NaverAuthService`, `KakaoAuthService`, `GoogleAuthService`)
  - 회원 조회/생성 및 JWT 발급 (`TokenService.loginWithMember`)
  - 토큰 쿠키 세팅 및 리다이렉트 응답
  - 공통 JWT 인증 필터 (`JwtAuthenticationFilter`)

### 2. 프론트엔드 소셜 로그인 플로우

#### 2-1. 로그인 진입 및 소셜 버튼

- 진입 화면: `Login.vue`
  - `SocialLoginButtons` 컴포넌트를 렌더링
  - `@click="handleSocialLogin"`을 통해 프로바이더(`'kakao' | 'google' | 'naver'`)를 전달받음
  - 현재 라우트의 `redirect` 쿼리(예: `/auth/login?redirect=/my/recipes`)를 읽어 **로그인 후 이동할 경로**를 계산하고, `localStorage('oauth_redirect')`에 저장
  - 이후 `openOAuthPopup(provider, clientId, redirectUri, additionalParams)` 호출

- `SocialLoginButtons.vue`
  - 카카오/구글/네이버 버튼을 렌더링하고, 클릭 시 `click` 이벤트로 프로바이더 키를 상위에 emit
  - UI 레벨에서만 역할 수행 (실제 OAuth 로직은 `openOAuthPopup`에 위임)

#### 2-2. OAuth 팝업 오픈 및 브리지 (`openOAuthPopup`)

- 위치: `frontend/src/utils/oauth.ts`
- `openOAuthPopup(provider, clientId, redirectUri, additionalParams?)`
  - 프로바이더별 설정(`authUrl`, `successType`, `errorType`, `windowName`)을 내부 `oauthConfigs`에서 조회
  - 랜덤 `state` 값 생성 후 `localStorage('{provider}_state')`에 저장
  - OAuth 인증 URL 쿼리 생성:
    - `response_type=code`
    - `client_id={clientId}`
    - `redirect_uri={redirectUri}` (프론트의 `/auth/{provider}/callback` URL)
    - `state={랜덤값}`
    - `additionalParams` (예: 구글의 `scope=openid email profile`)
  - `window.open`으로 팝업 오픈 (차단 시 안내 메시지)

- **부모 창 ↔ 팝업 창 통신**
  - 부모 창 측:
    - `window.addEventListener('message', handleMessage)` 등록
    - 메시지의 `origin`을 검사하여 동일 오리진 또는 localhost 개발 환경만 수용
    - 메시지 타입이 `*_LOGIN_SUCCESS`이면:
      - `authStore.login()` 호출 → 내부적으로 `/api/auth/refresh` + `/api/member/me` 연계
      - `localStorage('oauth_redirect')`를 읽어 라우터 이동 또는 `window.location.href`로 강제 이동
    - 메시지 타입이 `*_LOGIN_ERROR`이면:
      - 오류 메시지 alert 후 `oauth_redirect` 제거
  - `postMessage`가 실패하는 경우를 대비해:
    - 콜백 창에서 `localStorage('oauth_callback_event')` 등에 메시지를 기록
    - 부모 창은 `setInterval`로 localStorage를 폴링하여 동일 메시지 처리
  - 팝업이 닫혔는지 polling 하며, 일정 시간 후 리스너/인터벌 정리 및 타임아웃 처리

#### 2-3. 프론트 OAuth 콜백 라우트 및 콜백 화면

- 라우터 정의: `router/index.ts`
  - `/auth/naver/callback` → `NaverCallback.vue`
  - `/auth/google/callback` → `GoogleCallback.vue`
  - `/auth/kakao/callback` → `KakaoCallback.vue`

- 실제 콜백 화면: `OAuthCallback.vue`
  - `NaverCallback/GoogleCallback/KakaoCallback`에서 `provider`만 다르게 넘겨 재사용
  - 마운트 시 URL 쿼리 파라미터에서 `success`, `error`를 읽어 상태 판단
    - `success=true` → `handleSuccess()`
    - `success=false` 또는 `error` 존재 → `handleError(errorMessage)`
  - `handleSuccess`:
    - `window.opener.postMessage({ type: '{PROVIDER}_LOGIN_SUCCESS' }, window.location.origin)`
      시도
    - 실패 가능성(크로스 도메인, 팝업 차단 등)에 대비해 동일 메시지를 `localStorage`에 기록
    - 메시지를 여러 번 재전송 (100, 200, 300, 500, 1000ms 간격)
    - 1.5초 후 `window.close()`로 팝업 종료 시도
  - `handleError`:
    - 위와 동일하되, 타입만 `*_LOGIN_ERROR` + `error` 메시지 포함

#### 2-4. 인증 상태 관리 (`authStore`, 전역 가드)

- `authStore.ts`
  - `isAuthenticated`, `memberInfo`, `isInitialized` 상태 보유
  - `checkAuth()`:
    - `/api/auth/refresh` 호출 (`refreshToken()` → `authApi`)
    - 성공 시:
      - `isAuthenticated=true`
      - `loadMemberInfo()`로 `/api/member/me` 호출하여 `memberInfo` 채움
    - 실패 시 로그인 해제 상태로 초기화
  - `login()`:
    - 별도 토큰 저장 없이 `isAuthenticated=true`로 세팅 후 `loadMemberInfo()` 호출
  - `logout()`:
    - `/api/auth/logout` 호출 후, 스토어 상태만 초기화 (쿠키는 백엔드에서 삭제)

- 라우터 전역 가드 (`router.beforeEach`)
  - 최초 진입 시 `authStore.isInitialized`가 `false`이면 `checkAuth()` 한 번 실행
  - `meta.requiresAuth` 또는 `meta.requiresAdmin` 있는 라우트로 진입 시:
    - 로그인 안 되어 있으면 `/auth/login`으로 리다이렉트하고, 쿼리로 `redirect`에 원래 경로 저장
  - `requiresAdmin`인데 `isAdmin`이 아니라면 `/error/access`로 이동

### 3. 백엔드 소셜 로그인 플로우

#### 3-1. 콜백 엔드포인트 (`AuthController`)

- 클래스: `AuthController` (`backend/auth-service/.../web/AuthController.java`)
- 기본 경로: `/api/auth`
- 주요 엔드포인트:
  - `GET /api/auth/naver/callback?code&state`
  - `GET /api/auth/google/callback?code`
  - `GET /api/auth/kakao/callback?code`
- 공통 흐름:
  1. 프로바이더별 서비스(`NaverAuthService`, `GoogleAuthService`, `KakaoAuthService`) 호출해 로그인 프로세스 실행 → `TokenResponse` 획득
  2. `buildSuccessResponse(redirectUrl, tokenResponse, addCOOP, addCOEP)` 호출
     - `CookieUtils`를 사용해 `accessToken`, `refreshToken` 쿠키 생성
     - `302 FOUND` + `Location: {frontendUrl}/auth/{provider}/callback?success=true`로 리다이렉트
     - 구글의 경우 `Cross-Origin-Opener-Policy`, `Cross-Origin-Embedder-Policy` 헤더 일부 추가
  3. 예외 발생 시 `getErrorRedirectResponse(frontendUrl + "/auth/{provider}/callback")`로
     - `302` 리다이렉트
     - 쿼리: `?success=false&error=...` 로 에러 메시지 전달

#### 3-2. 프로바이더별 로그인 서비스

- 공통 특징:
  - 각 서비스에서 외부 OAuth 서버와 통신 (`RestTemplate` 사용)
  - 액세스 토큰 발급 → 사용자 정보 조회 → `MemberRepository`로 회원 조회/생성 → `TokenService.loginWithMember(member)` 호출
  - 회원 조회 키는 **이메일 + SocialRole** 조합을 사용

- `GoogleAuthService`
  - 설정: `google.client.id`, `google.client.secret`, `google.redirect.uri`
  - 토큰 엔드포인트: `https://oauth2.googleapis.com/token`
  - 사용자 정보: `https://www.googleapis.com/oauth2/v2/userinfo`
  - 신규 회원 생성 시:
    - `name`, `nickname`, `email`, `profileImage`, `socialRole=GOOGLE`, `isActive=TRUE`

- `KakaoAuthService`
  - 설정: `kakao.client.id`, `kakao.client.secret`, `kakao.redirect.uri`
  - 토큰 엔드포인트: `https://kauth.kakao.com/oauth/token`
  - 사용자 정보: `https://kapi.kakao.com/v2/user/me`
  - `KakaoUserDTO`에서 전화번호/생년월일 등을 받아 포맷팅(`LoginFormatter`) 후 저장
  - 신규 회원 생성 시:
    - `name`, `nickname`, `phone`, `email`, `birth`, `profileImage`, `socialRole=KAKAO`, `isActive=TRUE`

- `NaverAuthService`
  - 설정: `naver.client.id`, `naver.client.secret`
  - 토큰 엔드포인트: `https://nid.naver.com/oauth2.0/token`
  - 사용자 정보: `https://openapi.naver.com/v1/nid/me` (response 필드 안쪽에서 유저 정보 파싱)
  - 신규 회원 생성 시:
    - `name`, `nickname`, `phone`, `email`, `birth`, `gender`, `profileImage`, `socialRole=NAVER`, `isActive=TRUE`

#### 3-3. 토큰 발급 및 관리 (`TokenService`)

- 핵심 메서드: `loginWithMember(Member member)`
  - 기존 `RefreshToken` 레코드가 있으면 삭제(동시 세션 제한 및 깔끔한 회전)
  - JWT Claims:
    - `socialRole`: `GOOGLE` / `NAVER` / `KAKAO`
    - `role`: `USER` or `ADMIN` (현재는 하드코딩된 관리자 이메일로 판별)
    - `username`: 회원 이름
  - `JwtTokenProvider`를 통해 Access/Refresh 토큰을 생성
  - Refresh 토큰을 DB(`RefreshTokenRepository`)에 저장
  - `TokenResponse`(accessToken, refreshToken, 만료 시간들) 반환

- `refreshAccessToken(refreshToken)`
  - Refresh 토큰 파싱 및 검증
  - 비관적 락(`findByTokenForUpdate`)으로 동시 요청 제어
  - 기존 Refresh 토큰 삭제 후, 다시 `loginWithMember(member)` 호출해 토큰 재발급

- `logout(accessToken, refreshToken)`
  - Access 토큰은 `TokenBlacklist`에 저장하여 서버 측 블랙리스트 관리
  - Refresh 토큰은 DB에서 삭제

#### 3-4. 공통 JWT 검증 (`JwtAuthenticationFilter`)

- 모든 요청에 대해 실행되는 스프링 시큐리티 필터 (OncePerRequestFilter)
- 예외: `/api/auth/refresh`는 AccessToken 없이 RefreshToken만 필요하므로 필터 스킵
- 흐름:
  1. `accessToken` 쿠키에서 토큰 추출
  2. `TokenBlacklistChecker`로 블랙리스트 여부 확인 (auth-service에서만 동작)
  3. 토큰 파싱 후 `subject`와 `role` 클레임으로 `Authentication` 생성 후 `SecurityContext`에 세팅
  4. 다양한 JWT 예외 상황에 대해 적절한 HTTP 상태/에러 코드(JSON)를 내려줌

### 4. 종합 플로우 (요약)

1. 사용자가 `/auth/login` 진입
2. 소셜 로그인 버튼 클릭 → `handleSocialLogin(provider)` → `localStorage('oauth_redirect')` 세팅 → `openOAuthPopup` 호출
3. 팝업에서 프로바이더 로그인 → OAuth 서버가 백엔드의 `/api/auth/{provider}/callback`으로 리다이렉트
4. 백엔드 콜백:
   - 외부 OAuth 토큰/유저 정보 조회 → 회원 조회/생성 → JWT 발급 → HttpOnly 쿠키 세팅
   - 프론트 콜백 URL(`/auth/{provider}/callback?success=true|false&error=...`)로 302 리다이렉트
5. 프론트 콜백 페이지(`OAuthCallback`)에서 쿼리 파라미터를 확인하여
   - 성공: `*_LOGIN_SUCCESS` 메시지를 부모 창에 postMessage / localStorage로 전달
   - 실패: `*_LOGIN_ERROR` 메시지 전달
6. 부모 창:
   - 성공 메시지 수신 시 `authStore.login()` → `/api/member/me` 조회로 상태 갱신
   - `localStorage('oauth_redirect')` 경로로 이동

### 5. 개선 및 점검 포인트

#### 5-1. 보안 및 표준 측면

- **state 검증 누락**
  - 프론트에서 `state`를 랜덤 생성하여 `localStorage('{provider}_state')`에 저장하지만,
    백엔드 콜백(`/api/auth/.../callback`)에서 해당 `state` 값을 검증하는 로직은 현재 보이지 않음.
  - **개선 제안**:
    - 각 서비스(`NaverAuthService`, `KakaoAuthService`, `GoogleAuthService`)에서 `state`를 함께 처리하도록 확장하거나,
    - 서버에서 `state` 값을 세션/서버 저장소에 보관 후 콜백에서 비교 검증(특히 네이버는 `state`를 필수로 요구함).

- **PKCE 지원 여부**
  - 현재는 일반 Authorization Code Grant만 사용하는 것으로 보이고, 모바일/공개 클라이언트 환경에선 PKCE를 사용하는 것이 권장됨.
  - **개선 제안**:
    - 구글/네이버/카카오 중 PKCE를 지원하는 프로바이더에 대해 `code_challenge`, `code_verifier`를 추가로 도입 검토.

- **관리자 판별 로직**
  - `TokenService.isAdminEmail`에서 특정 이메일(`testadmin@test.com`) 하드코딩으로 관리자 권한을 설정.
  - **개선 제안**:
    - `Member` 엔티티에 별도의 `role` 필드나 권한 테이블을 사용하여 DB 기반으로 권한을 관리하도록 변경.

#### 5-2. 프론트/백엔드 통신 구조

- **중복된 postMessage 처리 로직**
  - `AppTopbar.vue`에도 `handleOAuthMessage`가 있고, `openOAuthPopup` 내부에도 메시지 처리 로직이 존재해 역할이 다소 중첩됨.
  - 현재 구조상:
    - `openOAuthPopup`은 로그인 페이지에서의 팝업 흐름을 보다 강하게 제어(라우팅/새로고침 복구 포함)
    - `AppTopbar`는 전역 레이아웃에서 메시지를 수신해 리다이렉트 처리
  - **개선 제안**:
    - OAuth 메시지 처리 책임을 하나의 유틸/Composable(`useOAuthMessageHandler` 등)로 통합하고,
      `AppTopbar`와 로그인 페이지에서 동일 로직을 재사용하도록 리팩터링.

- **에러 UX 개선**
  - 현재 콜백에서 에러 발생 시 단순 alert 및 리다이렉트이므로, 사용자 입장에서 어떤 문제가 발생했는지 명확하지 않을 수 있음.
  - **개선 제안**:
    - `/auth/login`으로 돌아올 때 에러 코드/메시지를 쿼리나 상태로 넘기고,
      로그인 화면 상단에 토스트/알림 컴포넌트로 일관된 에러 UI를 제공.

#### 5-3. 테스트 로그인 / 운영 분리

- `AuthController`의 `/api/auth/test/login`, `/api/auth/test/accounts` 및
  `TokenService.loginWithTestAccount`, `getTestAccounts`는 테스트 계정 전용 로직이며,
  이메일 prefix `test`로 테스트 계정을 구분.
- `app.test-login.enabled` 플래그로 기능 활성화 여부를 제어하고 있지만,
  운영 환경에서의 설정/오용 위험을 항상 염두에 두어야 함.
- **개선 제안**:
  - 운영 배포용 프로파일에서는 `app.test-login.enabled=false`가 확실히 보장되는지 점검 (`application-*.yml`).
  - 필요 시 별도의 `dev` 전용 도메인이나 IP 제한을 통해 테스트 로그인 엔드포인트 접근 제어 강화.

#### 5-4. 설정 및 환경 변수 관리

- 프론트 `.env`에 정의되는 `VITE_*_CLIENT_ID`, `VITE_*_REDIRECT_URI`와
  백엔드의 `*.client.id`, `*.client.secret`, `*.redirect.uri`가 서로 대응해야 함.
- **점검 포인트**:
  - 프론트 리다이렉트 URI(`VITE_*_REDIRECT_URI`)가 실제 라우터 경로(`/auth/{provider}/callback`)와 정확히 일치하는지,
    그리고 해당 URL이 각 소셜 플랫폼 콘솔에도 동일하게 등록되어 있는지 확인.
  - `app.frontend.url` 설정이 실제 배포 도메인과 일치하는지와, HTTPS 스킴 사용 여부.

---

### 6. 요약

- 소셜 로그인은 **팝업 기반 OAuth 플로우 + HttpOnly JWT 쿠키 + Pinia 스토어** 조합으로 비교적 깔끔하게 분리되어 있다.
- 백엔드는 각 프로바이더 서비스에서 사용자 정보를 통일된 `Member` 엔티티로 수렴하고, `TokenService`가 JWT/리프레시 토큰 생성을 일원화한다.
- 전반적인 구조는 확장성과 보안 측면에서 무난하지만,
  - `state` 검증 추가,
  - 관리자 권한 관리 방식 개선,
  - OAuth 메시지 처리 로직 통합,
  - 에러 UX/운영-테스트 분리 강화
  등의 개선을 통해 안정성과 유지보수성을 더 높일 수 있다.

### 7. 적용 완료된 개선 사항 (업데이트 메모)

- **state 검증 적용**
  - 서버에서 HMAC 서명된 state를 발급하고, 콜백 시 서명·만료·프로바이더 일치를 검증하는 구조로 반영됨.
  - **백엔드**
    - `OAuthStateService`에서 `generateState(provider)` / `validateState(provider, state)` 제공.
    - `GET /api/auth/oauth/state?provider=naver|google|kakao` → state 발급 후 `{ state }` JSON 응답.
    - 각 콜백(`/api/auth/{provider}/callback`) 진입 시 `OAuthStateService.validateState(provider, state)`로 검증.
    - 설정 키: `app.oauth.state-secret` (환경 변수 `OAUTH_STATE_SECRET`로 주입).
  - **프론트엔드**
    - `authApi.getOAuthState(provider)`로 state 사전 발급.
    - `Login.vue`의 `handleSocialLogin`에서 state를 받아 `openOAuthPopup(..., state)`로 전달.
  - **운영 권장 사항**
    - 프로덕션/도커 환경에서는 `OAUTH_STATE_SECRET`을 32자 이상의 안전한 랜덤 문자열로 설정해야 한다.
